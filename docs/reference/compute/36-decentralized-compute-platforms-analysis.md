# 去中心化计算平台分析

> 面向 `pi-worker` 的平台选型框架：在可信度、功能性、成本、运维之间做平衡

---

## 1. 评价模型

定义 5 个核心评估维度（每项 1~5 分）：

1. **可验证性**：能否证明执行过程/结果
2. **可扩展性**：是否支持并发、复杂任务、异构任务类型
3. **生态成熟度**：文档、SDK、运营节点、故障案例
4. **治理友好性**：是否容易接入质押、仲裁、信誉
5. **集成成本**：与现有 `pi/dAgent` 链路贴合度

> 目标不是追求单项高分，而是看综合分与项目阶段匹配度。

---

## 2. 平台类型与适配情况

### 2.1 去中心化云算力（如 Akash/类似生态）
- 优点：按需扩展、可运行更“完整”的执行环境
- 限制：调度和作业提交通常偏“平台工程”重，首屏易用性一般
- 与 `pi-worker` 适配度：
  - 中：适合模型推理、构建型任务、离线批处理
  - 低：短时交互式任务、超高并发实时性要求

### 2.2 TEE / 安全执行平台（如 EigenCompute/Phala）
- 优点：执行环境可信 + 私有数据友好
- 限制：配置复杂、接入路径长、资源成本相对高
- 与 `pi-worker` 适配度：高（安全型任务）

### 2.3 链下证明链路（如 Truebit/Cartesi/类似）
- 优点：可提供可验证性与仲裁闭环
- 限制：实现复杂度高、延迟可能更高
- 与 `pi-worker` 适配度：高（高价值任务）

### 2.4 平台化 serverless（如链上 function 生态）
- 优点：上手快、可观测性好
- 限制：受制于平台计费与限额
- 与 `pi-worker` 适配度：中（原型与轻量任务）

---

## 3. 平台选择建议（分阶段）

#### 原型期
- 以稳定性为先：优先 `serverless/受控平台`。
- 同步留好接口协议，方便后续替换。

#### 成长期
- 对关键任务逐步上 `去中心化算力平台`。
- 引入二阶验证：结果签名、异议上报、仲裁记录。

#### 成熟期
- 对高价值路径引入 `可验证执行`（TEE 或验证链路）
- 将平台依赖降到最低，通过抽象层解耦运行时。

---

## 4. 对 `dAgent` 的落地建议

- 不要一开始只追求“最去中心化”，也不要永远停在中心化平台。
- 建议用三层路由：
  1. `标准任务`：中心化低成本通道
  2. `关键任务`：去中心化算力通道
  3. `争议任务`：可验证/仲裁通道

---

## 5. 风险与对冲

- **供应商锁定风险**：通过统一任务协议和签名规范降低
- **运维复杂度风险**：保留最小化接口层 + 自动化部署模板
- **成本失控**：按任务类型预先打标签，做预算上限控制
- **治理风险**：预先定义仲裁标准与仲裁证据格式

---

## 6. 快速结论

当前阶段的最优路径通常是：

1. 先保留 `可控平台` 保障交付稳定性
2. 同步建立 `去中心化执行接口`
3. 关键路径逐步迁移到 `TEE/验证型` 平台
